## 一、开场锚点（30秒 · 三选一）

> 拿到"介绍一下你做过的项目"这道题时，用30秒完成"第一印象"。
> 不需要三种都背，选一种你说着最自然的。

### 风格A：好奇驱动型 ⭐ 推荐

> 适合：性格偏内向、做事认真的同学

"我之前做了一个全模态 AI 创作平台。起因是我在用各种AI工具生成图片、视频的时候，发现一个很有意思的问题——用户其实不关心用的是什么模型，他只想说一句话就拿到结果。所以我就想，能不能让 AI 自己来理解用户要什么、自己选最合适的模型？这其实就是一个 Agent 的设计问题。我围绕这个想法做了一个完整的系统，支持文生图、文生视频、语音合成这些多模态能力。"

**为什么这么说**：老师听到"有意思的问题"会觉得你有好奇心，听到"Agent的设计问题"会觉得你有抽象思考能力。

### 风格B：问题驱动型

> 适合：表达能力较强、喜欢讲故事的同学

"我做了一个 AI 创作平台，要解决的核心问题是：现在的 AI 生成工具门槛太高了，用户得自己选模型、调参数、写英文提示词。我就想，能不能设计一个智能 Agent，让用户说一句中文，系统自动完成从意图理解到模型选择到内容生成的全流程？这个过程中我深入学习了 Agent 架构设计、多模态 API 对接、异步任务管理这些技术。"

**为什么这么说**：开头直接点出"要解决的问题"，老师知道你有目标感，不是盲目做项目。

### 风格C：技术趋势型

> 适合：对 AI 前沿比较了解、适合学术型导师

"我关注到 Google 发布 Gemini 多模态能力、Veo 视频生成之后，就很想亲手试试怎么把这些前沿能力整合成一个真正可用的产品。所以我设计了一个全模态 AI 创作平台，核心做了一个 Agent，它能自动分析用户的自然语言需求，路由到最合适的生成模型。这个过程让我对 Agent 系统设计、提示词工程、多模态技术都有了比较深入的理解。"

**为什么这么说**：展示你对前沿技术的敏感度，暗示你有持续学习的习惯，这对导师来说很重要。

### 3.1 Agent（智能代理）

🎯 **通俗比喻**：
Agent 就像一个聪明的前台接待。你去酒店说"我想要一间安静的房间"，前台不会让你自己翻房间列表，他会理解你的需求、查系统、给你推荐最合适的房间。Agent 做的就是同样的事——理解需求、做决策、安排执行。

📖 **技术本质**：
Agent 是一个具备"感知 → 推理 → 行动"能力的智能系统。和传统的硬编码逻辑不同，Agent 使用 LLM（大语言模型）作为推理引擎，能处理模糊的、非结构化的输入，动态决定执行路径。在我的项目里，Agent 包含意图分析、路由决策、提示词增强三个核心组件，形成一个四层处理管道。

💭 **我的理解**：
"我觉得 Agent 最核心的价值不是'它能做事'，而是'它能帮用户做决定'。传统系统是把决策权交给用户——你选模型、你写提示词、你调参数。Agent 是把这个决策负担从用户身上拿走。我做这个项目的过程中越来越觉得，好的 Agent 设计不是让 AI 做更多事，而是让用户做更少决定。"

❓ **可能追问**：
- **"Agent 和普通的 if-else 判断有什么区别？"**
  → 回答思路：if-else 是穷举式的，你要预想所有情况；Agent 用 LLM 做推理，能处理意料之外的输入。比如用户说"帮我搞个动图"，规则引擎很难判断这是图片还是视频，但 LLM 能理解这其实是想要视频。
- **"你的 Agent 能自主学习吗？"**
  → 回答思路：目前不能，是基于预设规则和 LLM 推理的。但如果加入用户反馈循环，比如用户对生成结果打分，可以逐步优化 Prompt 策略。这也是我想在研究生阶段深入探索的方向。

### 3.2 意图分析 / 意图识别

🎯 **通俗比喻**：
就像餐厅服务员听你说"来个不辣的"，他不只是听到了"不辣"这个词，还得判断你要菜还是汤、几个人吃、大概什么价位。意图分析就是从用户的一句话里"读出"他到底想干什么。

📖 **技术本质**：
意图分析（Intent Analysis）是 NLP 领域的经典任务，目标是从非结构化的自然语言中提取结构化信息。在我的项目中，IntentAnalyzer 通过精心设计的 System Prompt 引导 gpt-4o-mini，从用户输入中提取四类信息：内容类型（IMAGE/VIDEO/AUDIO）、意图场景（text_to_image 等）、技术参数（宽高比、时长、分辨率）、清洗后的纯创意提示词。输出是严格的 JSON 格式，通过 Jackson 反序列化为 AnalyzedIntent 对象。

💭 **我的理解**：
"意图分析这块我踩了一些坑。最开始我考虑过用正则和关键词匹配，比如看到'视频'就判定为VIDEO。但用户说的话千变万化——'帮我搞个动画'、'让这张图动起来'、'做个短片'，规则引擎根本cover不住。用 LLM 之后效果好很多，但又有新问题：LLM 返回的 JSON 有时候会被 Markdown 代码块包裹，有时候格式不完全对。所以我做了一层清洗逻辑，专门处理这种不确定性。这让我意识到，用 LLM 做工程化的事情，容错和解析的 robustness 非常重要。"

❓ **可能追问**：
- **"LLM 做意图分析的准确率怎么保证？"**
  → 回答思路：三个手段。一是 Prompt 设计得很详细，枚举了所有合法值和示例；二是 temperature 设为 0.1，降低随机性；三是有参考图片时会做规则修正，比如有图片且类型是IMAGE，就强制调整为 image_to_image。
- **"为什么不用传统的 NLU 模型来做？"**
  → 回答思路：传统 NLU 需要标注数据和训练，成本高、周期长。LLM 的优势是零样本能力，通过 Prompt 就能定义解析规则，迭代非常快。在项目初期或数据不足的情况下，这种方案性价比最高。

### 3.3 多模态（Multimodal）

🎯 **通俗比喻**：
人类天生就是"多模态"的——你能同时看图、听声音、读文字、说话。早期的 AI 是"偏科生"，一个模型只能处理一种数据。现在的多模态 AI 更像一个"全能选手"，能同时理解和生成文字、图片、视频、声音。

📖 **技术本质**：
多模态指模型能够处理和生成不同类型（模态）的数据——文本、图像、音频、视频等。比如 Google Gemini 可以接收文本+图片输入，输出文本或图片；Veo 可以接收文本（或文本+图片）输入，输出视频。技术上，多模态模型通常在统一的表示空间中对齐不同模态的特征，使得跨模态的理解和生成成为可能。

💭 **我的理解**：
"做这个项目之后我对多模态最深的感受是——从应用层来看，不同模态之间的 API 协议差异巨大。图片生成是同步的，调一个 API 直接拿结果；视频生成是异步的，要用长时运行操作模式，每10秒轮询一次；音频又不一样。所以多模态不只是模型层面的挑战，在工程层面也需要一个好的抽象来统一处理这些差异，这就是我做 Provider 抽象体系的原因。"

❓ **可能追问**：
- **"不同模态的数据在模型内部是怎么融合的？"**
  → 回答思路：主流方法是把不同模态的数据都转换到一个共享的向量空间（embedding space），让文本和图像的表示可以相互对齐。比如 CLIP 就是通过对比学习让文本和图像的 embedding 靠近。不过作为应用开发者，我更关注的是 API 层面怎么使用这种能力。
- **"你觉得多模态的瓶颈在哪？"**
  → 回答思路：我觉得目前最大的瓶颈是生成速度和一致性。视频生成还需要几分钟，实时交互做不到；而且多模态间的一致性很难保证，比如图片风格和文字描述可能对不上。这些都是值得研究的方向。

### 3.4 Diffusion 扩散模型原理

🎯 **通俗比喻**：
想象你有一张清晰的照片，你一点一点往上面撒沙子，最后变成一堆沙子，什么都看不出来——这就是"加噪声"的过程。扩散模型学会了一个本事：给它一堆沙子（纯噪声），它能根据你的描述（Prompt），一点一点把沙子扫掉，还原出一张和描述匹配的图片。它并不是"画画"，而是"从混沌中恢复秩序"。

📖 **技术本质**：
Diffusion Model 的核心是两个过程。前向过程（Forward Process）：对训练图片逐步添加高斯噪声，经过 T 步后变成纯噪声。反向过程（Reverse Process）：训练一个去噪网络（通常是 U-Net），学习每一步如何从带噪声的图片中恢复信号。生成时，从纯噪声出发，通过文本 Prompt 的条件引导（Classifier-Free Guidance），逐步去噪得到目标图片。

💭 **我的理解**：
"我觉得扩散模型最反直觉的地方是——它不是像人一样'从白纸上画画'，而是'从一团乱麻中雕刻'。这个思路和传统的生成模型（比如 GAN 直接从潜向量映射到图片）完全不同。从应用角度看，Diffusion 的好处是生成质量非常高、训练稳定，缺点是推理慢（需要多步去噪）。这也解释了为什么我项目中图片生成需要 10-30 秒——它在背后做了很多步去噪迭代。"

❓ **可能追问**：
- **"Diffusion 和 GAN 有什么区别？"**
  → 回答思路：GAN 是两个网络对抗——生成器造假、判别器鉴别，训练不稳定容易崩溃。Diffusion 是单一网络逐步去噪，训练更稳定，生成质量更高，但推理速度慢。目前主流趋势是 Diffusion 逐渐替代 GAN。
- **"你的项目直接用了 Diffusion 吗？"**
  → 回答思路：我是通过 Google Gemini 的 API 来调用的，底层使用了扩散模型技术。作为应用开发者，我不需要自己训练模型，但理解底层原理帮助我更好地设计 Prompt 和理解生成行为，比如为什么某些细节描述能显著提升质量。

### 3.5 Prompt Engineering（提示词工程）

🎯 **通俗比喻**：
就像你跟一个画师约稿。你说"画只猫"，画师可能画出各种猫。但你说"画一只橘色的短毛猫，坐在窗台上，夕阳光照进来，水彩画风格"，画师就知道你要什么了。Prompt Engineering 就是学会怎么"精准描述需求"让 AI 输出你想要的结果。

📖 **技术本质**：
Prompt Engineering 是通过精心设计输入文本来引导 LLM 行为的技术。核心技巧包括：明确角色设定（System Prompt 定义 AI 身份）、结构化指令（定义输出格式和约束）、Few-shot 示例（提供参考样例）、参数控制（temperature 控制随机性）。在我的项目中有两处应用：一是 IntentAnalyzer 的系统提示词设计，引导 LLM 稳定输出结构化 JSON；二是 PromptEnhancer 自动增强用户的创作描述。

💭 **我的理解**：
"Prompt Engineering 看起来就是'写几句话'，但实际上它是一种'用自然语言编程'。我在设计 IntentAnalyzer 的 Prompt 时，迭代了很多版本。最开始写得太简单，LLM 返回的格式经常不对；后来我加了详细的参数枚举、默认值规则、具体示例，稳定性才上来。我的体会是，好的 Prompt 像好的 API 文档——要定义清楚输入输出、边界情况和默认行为。"

❓ **可能追问**：
- **"temperature 参数是什么？怎么设的？"**
  → 直接引导到下一个深潜知识点 3.8
- **"Few-shot 和 Zero-shot 有什么区别？"**
  → 回答思路：Zero-shot 是不给例子，直接让模型做任务；Few-shot 是给几个输入-输出的例子，让模型学着做。我的意图分析用的是 Zero-shot + 详细指令的方式，因为我发现对 gpt-4o-mini 来说，清晰的格式定义比给例子更有效。

### 3.6 Provider 抽象与开闭原则

🎯 **通俗比喻**：
就像家里的插座——不管你插电视、电脑还是手机充电器，插座的接口是一样的。Provider 抽象就是定义一个"标准接口"，不管背后是 Google 的模型还是 OpenAI 的模型，只要"插上去"就能用。将来换一个新的模型提供商，不需要把家里的电线全拆了重新布，只要新电器适配同样的插头就行。

📖 **技术本质**：
开闭原则（Open-Closed Principle）是 SOLID 设计原则之一：对扩展开放，对修改关闭。在我的项目中，定义了 ContentProvider 基接口和三个子接口（ImageGenerationProvider / VideoGenerationProvider / AudioGenerationProvider），每个子接口定义了特定模态的能力方法。具体实现类（如 GoogleImageProvider）通过 @Component 注解注册到 Spring 容器，ProviderRouter 通过依赖注入自动收集所有实现，根据配置文件的 active-provider 选择激活哪一个。

💭 **我的理解**：
"这个设计是我在项目里花心思最多的部分之一。一开始我把 Google API 的调用逻辑直接写在 Service 里，后来发现如果想加 OpenAI 的 DALL-E，就得改一大堆代码。抽出 Provider 接口之后，加新模型变成了三步：写实现类、加注解、改配置文件。这个过程让我真正理解了'面向接口编程'不是课本上的口号，而是解决实际问题的工具。"

❓ **可能追问**：
- **"Spring 的依赖注入是怎么自动发现 Provider 的？"**
  → 回答思路：Spring 在启动时会扫描所有带 @Component 注解的类，如果一个类实现了 ImageGenerationProvider 接口，它就会被注入到 ProviderRouter 中声明的 `List<ImageGenerationProvider>` 里。ProviderRouter 遍历列表，匹配 active-provider 配置来选择。
- **"除了开闭原则，你还用到了什么设计原则？"**
  → 回答思路：单一职责（IntentAnalyzer 只做意图分析，PromptEnhancer 只做增强）、依赖倒置（Service 依赖 Provider 接口而非具体实现）、关注点分离（Agent 四层各司其职）。

### 3.7 异步任务与轮询机制

🎯 **通俗比喻**：
就像你去餐厅点菜。你不会站在厨房门口等厨师做完——你点完菜拿到一个号码牌，回座位等着，偶尔看看叫号屏。服务员在后厨忙活，做好了就更新叫号屏。这就是"异步任务 + 轮询"——你发一个请求，拿到一个任务 ID（号码牌），然后定期去查状态（看叫号屏）。

📖 **技术本质**：
在我的项目中，AI 内容生成耗时很长（图片 10-30s，视频 1-5min），不能让 HTTP 请求同步阻塞。采用的方案是：后端用 Spring 的 @Async 注解将生成任务放到独立线程池执行，主请求立即返回 taskId；前端拿到 taskId 后每秒调用一次 GET /api/aigc/task/{taskId} 查询状态。任务状态经历 PENDING → PROCESSING → COMPLETED/FAILED 的流转。视频生成还有"双重轮询"：前端轮询后端任务状态，后端内部每10秒轮询 Google Veo 的操作状态。

💭 **我的理解**：
"异步任务这块是整个项目工程难度最大的部分。有几个坑值得说：@Async 方法不能在同一个类中直接调用，否则注解会失效——Spring 的 AOP 代理只对外部调用生效。另外，异步线程中的异常不会抛到主线程，必须在 try-catch 里手动更新任务状态为 FAILED，不然任务会永远卡在 PROCESSING。这些都是实际开发中才会遇到的问题。"

❓ **可能追问**：
- **"轮询有什么缺点？有更好的方案吗？"**
  → 回答思路：轮询的缺点是浪费请求——180次轮询中可能170次拿到的都是 PROCESSING。更好的方案是 WebSocket 或 SSE（Server-Sent Events），服务端主动推送状态变化。我目前用轮询是因为实现简单，如果要优化可以改成 SSE。
- **"@Async 的线程池是怎么配置的？"**
  → 回答思路：Spring Boot 默认有一个 SimpleAsyncTaskExecutor，但生产环境建议自定义 ThreadPoolTaskExecutor，设置核心线程数、最大线程数、队列容量，避免线程耗尽。

### 3.8 Temperature 参数

🎯 **通俗比喻**：
Temperature 就像"创意旋钮"。旋到最低（接近0），AI 就像一个一丝不苟的会计，每次都给最确定的回答，回答也很稳定；旋到最高（接近1或更高），AI 就像一个天马行空的诗人，回答更有创意但也更不可预测。

📖 **技术本质**：
Temperature 控制的是 LLM 输出的概率分布。在模型预测下一个 token 时，会计算每个候选 token 的概率。Temperature 低时，概率分布变得"尖锐"，高概率 token 被选中的概率更大，输出更确定；Temperature 高时，分布变得"平坦"，低概率 token 也有机会被选中，输出更多样。数学上，是在 softmax 函数中除以 temperature 值再做归一化。

💭 **我的理解**：
"在我的项目里，意图分析的 temperature 设为 0.1——因为我需要确定性的结构化输出，同一句话每次解析的结果应该一致。如果 temperature 设高了，可能同一句'画一只猫'有时被解析成 IMAGE，有时被解析成 VIDEO，系统就不稳定了。但如果是做创意内容生成的 Prompt，temperature 反而应该高一些，鼓励多样性。关键是要理解你的场景需要确定性还是创造性。"

❓ **可能追问**：
- **"Temperature 为 0 是完全确定的吗？"**
  → 回答思路：理论上 temperature=0 是贪婪解码，每次选概率最大的 token，输出应该一致。但实际工程中因为浮点精度等原因，并不保证100%一致，所以通常设一个很小的值比如 0.1。
- **"除了 temperature，还有什么参数影响输出？"**
  → 回答思路：常见的还有 top_p（核采样，限制候选 token 的累积概率）、top_k（限制候选 token 数量）、max_tokens（限制输出长度）。在我的意图分析中，主要依赖 temperature 和结构化 Prompt 来控制输出质量。

## 三、深潜知识点（工程向）

> 除了复试版的8个知识点外，以下是求职面试重点追问的工程技术点。

---

### 3.1 Spring Boot @Async 异步执行

🎯 **通俗比喻**：
@Async 就像你给员工一个任务说"你去处理这件事，完了更新到看板上，我先去做别的了"。主线程不等你，另起一个线程去干活。

📖 **技术本质**：
Spring 的 @Async 注解通过 AOP 代理实现。加了 @Async 的方法被调用时，Spring 不会在当前线程执行，而是提交到 TaskExecutor 线程池异步执行。需要在启动类加 @EnableAsync 开启。返回值可以是 void（fire-and-forget）或 Future/CompletableFuture（需要结果时）。

💭 **我的理解**：
"用 @Async 踩了两个坑。第一个，同类内部调用不生效——因为 Spring AOP 是基于代理对象的，类内部 this 调用绕过了代理。第二个，异步线程的异常不会传播到调用者——所以必须在异步方法内部 try-catch，手动把错误状态写到数据库。这两个问题让我对 Spring AOP 的原理理解深了很多。"

❓ **可能追问**：
- **"默认的线程池有什么问题？"** → SimpleAsyncTaskExecutor 每次新建线程，没有复用，生产环境要换 ThreadPoolTaskExecutor。
- **"异步方法的事务怎么处理？"** → 异步方法运行在新线程，和调用者不在同一个事务。需要独立的 @Transactional。

---

### 3.2 OkHttp 调用外部 API

🎯 **通俗比喻**：
OkHttp 是一个专业的快递员——支持自动重连、连接池复用、代理中转，比邮局（RestTemplate）更灵活。

📖 **技术本质**：
OkHttp 是 Square 开源的 HTTP 客户端，特点是连接池管理、拦截器链、支持 HTTP/2。在我的项目中用它调 Google API，因为需要处理 Base64 大文件传输、代理配置、长超时（视频生成 readTimeout 600秒）。通过 `OkHttpClient.Builder` 配置代理 `proxy()`、超时 `readTimeout()`、连接池 `connectionPool()`。

💭 **我的理解**：
"选 OkHttp 不是因为它'更好'，是因为场景匹配。Google API 返回的图片是 Base64 编码的二进制数据，OkHttp 的 ResponseBody.bytes() 处理更直接。代理配置也更简洁，一行 `builder.proxy(new Proxy(...))` 就搞定了。RestTemplate 调 OneRouter 反而更合适，因为就是标准的 JSON 交互，没必要引入额外复杂度。"

---

### 3.3 RESTful API 设计

🎯 **通俗比喻**：
REST 是一种"给接口取名字的规范"。就像图书馆——GET /books 是看书单，POST /books 是新进一本书，DELETE /books/123 是下架123号书。动词靠 HTTP 方法表达，名词靠 URL 路径表达。

📖 **技术本质**：
我的项目中 API 设计遵循 REST 规范：
- `POST /api/aigc/generate` — 创建生成任务（资源创建用 POST）
- `GET /api/aigc/task/{taskId}` — 查询任务状态（资源查询用 GET）
- `GET /api/aigc/gallery` — 获取灵感广场列表
- `POST /api/aigc/gallery/save` — 发布作品到广场
- `GET /api/aigc/assets` — 获取我的资产列表
- `DELETE /api/aigc/assets/{id}` — 删除资产

💭 **我的理解**：
"API 设计看起来简单，但有几个细节值得注意。generate 接口用 POST 而不是 GET，因为它有副作用（创建任务记录）。task/{taskId} 用 GET 因为它是幂等的，重复调用不会改变状态——这对前端轮询很重要，网络抖动导致重复请求不会出问题。"

---

### 3.4 轮询 vs WebSocket vs SSE

🎯 **通俗比喻**：
- **轮询**：你每分钟看一次手机有没有新消息（主动查）
- **WebSocket**：你和朋友打电话，双方可以随时说话（双向实时通道）
- **SSE**：你订阅了一个新闻推送，有新消息它主动推给你（单向推送）

📖 **技术本质**：

| 特性 | 轮询 | WebSocket | SSE |
|------|------|-----------|-----|
| 方向 | 客户端 → 服务端 | 双向 | 服务端 → 客户端 |
| 连接 | 短连接，每次新建 | 长连接 | 长连接 |
| 开销 | 大量无效请求 | 维护连接状态 | 中等 |
| 实现复杂度 | 低 | 高 | 中 |
| 适用场景 | 状态查询 | 实时聊天 | 进度推送 |

💭 **我的理解**：
"我选轮询是因为实现简单、和 REST 架构一致、客户端断线自动恢复。如果优化的话，SSE 最合适——任务状态推送是单向的，不需要 WebSocket 的全双工能力。SSE 用 Spring 的 SseEmitter 就能实现，前端用 EventSource API，比 WebSocket 轻量得多。"

---

### 3.5 策略模式 / 工厂模式在 Provider 中的应用

🎯 **通俗比喻**：
- **策略模式**：就像导航软件——输入目的地一样，但可以选"最快路线""最短路线""避开高速"，不同策略用不同算法。
- **工厂模式**：就像自动贩卖机——你按图片按钮出图片生成器，按视频按钮出视频生成器，你不需要知道机器里面怎么造的。

📖 **技术本质**：
在我的项目中，RoutingAgent 的模型选择用了策略模式——switch 表达式根据 ContentType 分发到不同的选择策略（selectImageModel / selectVideoModel / selectAudioModel），每种策略的选择逻辑独立。ProviderRouter 的 Provider 选择类似工厂模式——通过 Spring 依赖注入收集所有实现，根据配置返回激活的 Provider 实例。

💭 **我的理解**：
"我没有刻意去'用设计模式'，而是在解决'怎么让代码好扩展'这个问题时，发现最自然的解法就是这些模式。比如一开始图片和视频的生成逻辑写在一起，后来发现每加一个模态就要改很多地方，很痛苦。抽出接口之后，加新模态就是'实现接口 + 注册'两步，自然就成了策略模式和工厂模式的结合。"

---

### 3.6 错误处理与降级

🎯 **通俗比喻**：
就像飞机有两个引擎——一个挂了还能飞。好的系统不是"永远不出错"，而是"出错了还能体面地应对"。

📖 **技术本质**：
我的项目中错误处理分三层。Provider 层：每个 generate() 方法内部 try-catch，失败返回 GenerationResult.failure()，不抛异常。Service 层：executeGenerationAsync() 检查 result.isSuccess()，失败更新 FAILED 状态并记录 errorMessage。外层兜底：大 try-catch 防止任何未预期异常导致任务卡在 PROCESSING。

💭 **我的理解**：
"AI 应用的错误处理比传统应用复杂很多。传统应用报错通常是确定的——数据库连接断了、参数格式不对。AI 应用的'错误'可能是模型触发了安全过滤、生成结果质量太差、API 超时。我的处理原则是：可重试的自动重试，不可重试的快速失败并告知用户原因。如果继续优化，可以加熔断器（Circuit Breaker），OneRouter 挂了自动降级到规则引擎做意图分析。"

## 五、故事线自洽包（求职版调整）

### 5.1 为什么做这个项目

> "异步方法在新线程执行，和主线程不共享事务上下文。一开始在 generate() 里创建任务后调异步方法，异步方法里查任务发现查不到——因为主线程的事务还没提交。后来把创建和异步调用放在事务提交后执行，问题才解决。"

> "调 Gemini 图片 API 默认 10 秒超时经常超时失败。调 Veo 视频 API 需要等几分钟。我给不同场景配了不同的超时——图片 readTimeout 120 秒，视频 readTimeout 600 秒。这是通过不同的 OkHttpClient 实例来实现的。"

### 5.3 被问"为什么用 Java 而不是 Python"

> "AI 应用确实很多用 Python 做，因为 AI 生态更丰富。但我选 Java 是因为：第一，Spring Boot 的企业级特性（依赖注入、AOP、异步支持）让后端架构设计更规范；第二，Java 的类型系统对大项目的可维护性更好；第三，我更熟悉 Java 生态。如果要做模型训练和微调，Python 确实更合适，但做 AI 应用的后端服务，Java 完全没有问题。"
