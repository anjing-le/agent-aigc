深潜知识点

每个概念四层递进：比喻→技术本质→我的理解→可能追问
1
Agent（智能代理）
▼
🎯 通俗比喻

Agent 就像一个聪明的前台接待。你去酒店说"我想要一间安静的房间"，前台不会让你自己翻房间列表，他会理解你的需求、查系统、给你推荐最合适的房间。Agent 做的就是同样的事——理解需求、做决策、安排执行。

📖 技术本质

Agent 是一个具备"感知→推理→行动"能力的智能系统。与硬编码逻辑不同，Agent 使用 LLM 作为推理引擎，能处理模糊的、非结构化的输入，动态决定执行路径。在本项目中，Agent 包含意图分析、路由决策、提示词增强三个核心组件，形成四层处理管道。

💭 我的理解（第一人称）
"我觉得 Agent 最核心的价值不是'它能做事'，而是'它能帮用户做决定'。传统系统是把决策权交给用户——你选模型、你写提示词、你调参数。Agent 是把这个决策负担从用户身上拿走。我做这个项目的过程中越来越觉得，好的 Agent 设计不是让 AI 做更多事，而是让用户做更少决定。"
❓ 可能追问
"Agent 和普通的 if-else 判断有什么区别？"
→ if-else 是穷举式的，要预想所有情况；Agent 用 LLM 做推理，能处理意料之外的输入。比如用户说"帮我搞个动图"，规则引擎很难判断是图片还是视频，但 LLM 能理解语义。
"你的 Agent 能自主学习吗？"
→ 目前不能，是基于预设规则和 LLM 推理的。但如果加入用户反馈循环，可以逐步优化 Prompt 策略。这也是想在研究生阶段深入探索的方向。
2
意图分析 / 意图识别
▼
🎯 通俗比喻

就像餐厅服务员听你说"来个不辣的"，他不只是听到了"不辣"这个词，还得判断你要菜还是汤、几个人吃、大概什么价位。意图分析就是从用户的一句话里"读出"他到底想干什么。

📖 技术本质

IntentAnalyzer 通过精心设计的 System Prompt 引导 gpt-4o-mini，从用户输入中提取四类信息：内容类型（IMAGE/VIDEO/AUDIO）、意图场景（text_to_image 等）、技术参数（宽高比、时长、分辨率）、清洗后的纯创意提示词。输出严格 JSON 格式，通过 Jackson 反序列化为 AnalyzedIntent 对象。

💭 我的理解
"意图分析这块我踩了一些坑。最开始考虑过用正则和关键词匹配，但用户说的话千变万化——'帮我搞个动画'、'让这张图动起来'、'做个短片'，规则引擎根本cover不住。用 LLM 之后又有新问题：返回的 JSON 有时候被 Markdown 代码块包裹。所以我做了一层清洗逻辑，这让我意识到，用 LLM 做工程化的事情，容错和 robustness 非常重要。"
❓ 可能追问
"LLM 做意图分析的准确率怎么保证？"
→ 三个手段：Prompt 详细枚举所有合法值和示例；temperature 设 0.1 降低随机性；有参考图片时做规则修正（强制 image_to_image）。
"为什么不用传统 NLU 模型？"
→ 传统 NLU 需要标注数据和训练，成本高周期长。LLM 的零样本能力通过 Prompt 就能定义解析规则，迭代非常快，性价比最高。
3
多模态（Multimodal）
▼
🎯 通俗比喻

人类天生就是"多模态"的——能同时看图、听声音、读文字、说话。早期的 AI 是"偏科生"，一个模型只能处理一种数据。现在的多模态 AI 更像一个"全能选手"，能同时理解和生成文字、图片、视频、声音。

📖 技术本质

多模态指模型能够处理和生成不同类型（模态）的数据。Gemini 可以接收文本+图片输入，输出文本或图片；Veo 可以接收文本或文本+图片输入，输出视频。技术上，多模态模型通常在统一的表示空间中对齐不同模态的特征，使跨模态的理解和生成成为可能。

💭 我的理解
"做这个项目之后我对多模态最深的感受是——从应用层来看，不同模态之间的 API 协议差异巨大。图片是同步的，视频是异步的，音频又不一样。所以多模态不只是模型层面的挑战，在工程层面也需要一个好的抽象来统一处理差异，这就是我做 Provider 抽象体系的原因。"
❓ 可能追问
"不同模态的数据在模型内部是怎么融合的？"
→ 主流方法是把不同模态的数据都转换到共享的向量空间（embedding space），让文本和图像表示相互对齐。比如 CLIP 通过对比学习实现这一点。
4
Diffusion 扩散模型原理
▼
🎯 通俗比喻

想象你有一张清晰的照片，你一点一点往上面撒沙子，最后变成一堆沙子——这就是"加噪声"。扩散模型学会了一个本事：给它一堆沙子（纯噪声），它能根据你的描述（Prompt），一点一点把沙子扫掉，还原出匹配描述的图片。它不是"画画"，而是"从混沌中恢复秩序"。

📖 技术本质

前向过程：对训练图片逐步添加高斯噪声，经过 T 步变成纯噪声。反向过程：训练去噪网络（通常 U-Net），学习每步如何恢复信号。生成时从纯噪声出发，通过文本 Prompt 的条件引导（Classifier-Free Guidance）逐步去噪得到目标图片。

💭 我的理解
"我觉得扩散模型最反直觉的地方是——它不是像人一样'从白纸上画画'，而是'从一团乱麻中雕刻'。从应用角度看，Diffusion 的好处是生成质量高、训练稳定，缺点是推理慢（多步去噪）。这也解释了为什么项目中图片生成需要 10-30 秒。"
❓ 可能追问
"Diffusion 和 GAN 有什么区别？"
→ GAN 是两个网络对抗，训练不稳定容易崩溃。Diffusion 是单一网络逐步去噪，训练更稳定，质量更高，但推理速度慢。目前趋势是 Diffusion 逐渐替代 GAN。
5
Prompt Engineering（提示词工程）
▼
🎯 通俗比喻

就像你跟画师约稿。你说"画只猫"，画师可能画出各种猫。但你说"画一只橘色短毛猫，坐在窗台上，夕阳光照进来，水彩画风格"，画师就知道了。Prompt Engineering 就是学会怎么"精准描述需求"让 AI 输出你想要的结果。

📖 技术本质

核心技巧包括：角色设定（System Prompt 定义 AI 身份）、结构化指令（定义输出格式和约束）、Few-shot 示例（提供参考样例）、参数控制（temperature 控制随机性）。项目中两处应用：IntentAnalyzer 的系统提示词设计，PromptEnhancer 自动增强创作描述。

💭 我的理解
"Prompt Engineering 看起来就是'写几句话'，但实际上它是一种'用自然语言编程'。我在设计 IntentAnalyzer 的 Prompt 时迭代了很多版本。最开始太简单，LLM 返回格式经常不对；后来加了详细的参数枚举、默认值规则、具体示例，稳定性才上来。我的体会是，好的 Prompt 像好的 API 文档——要定义清楚输入输出、边界情况和默认行为。"
❓ 可能追问
"Few-shot 和 Zero-shot 有什么区别？"
→ Zero-shot 不给例子直接让模型做；Few-shot 给几个输入-输出的例子让模型学。我的意图分析用 Zero-shot + 详细指令，因为清晰的格式定义比给例子更有效。
6
Provider 抽象与开闭原则
▼
🎯 通俗比喻

就像家里的插座——不管你插电视、电脑还是手机充电器，插座接口是一样的。Provider 抽象就是定义一个"标准接口"，不管背后是 Google 还是 OpenAI，只要"插上去"就能用。将来换新模型不需要拆电线，只要适配同样的插头。

📖 技术本质

定义 ContentProvider 基接口和三个子接口（Image/Video/AudioGenerationProvider）。具体实现类通过 @Component 注册到 Spring 容器，ProviderRouter 通过依赖注入自动收集所有实现，根据配置文件的 active-provider 选择激活哪一个。对扩展开放，对修改关闭。

💭 我的理解
"一开始我把 Google API 的调用逻辑直接写在 Service 里，后来发现想加 DALL-E 就得改一大堆代码。抽出 Provider 接口之后，加新模型变成三步：写实现类、加注解、改配置文件。这个过程让我真正理解了'面向接口编程'不是课本上的口号，而是解决实际问题的工具。"
❓ 可能追问
"Spring 的依赖注入是怎么自动发现 Provider 的？"
→ Spring 启动时扫描所有 @Component 类，实现了 ImageGenerationProvider 的就被注入到 ProviderRouter 的 List 里。ProviderRouter 遍历匹配 active-provider 配置来选择。
7
异步任务与轮询机制
▼
🎯 通俗比喻

就像去餐厅点菜。你不会站在厨房门口等——点完菜拿到号码牌，回座位等着，偶尔看看叫号屏。"异步任务+轮询"就是发一个请求拿到任务ID（号码牌），然后定期查状态（看叫号屏）。

📖 技术本质

后端用 @Async 将生成任务放到独立线程池，主请求立即返回 taskId。前端每秒 GET /api/aigc/task/{taskId} 查询状态。任务经历 PENDING→PROCESSING→COMPLETED/FAILED。视频有"双重轮询"：前端轮询后端，后端内部每10秒轮询 Google Veo 的 operation 状态。

💭 我的理解
"异步任务是工程难度最大的部分。两个坑：@Async 方法不能在同类中直接调用（Spring AOP 代理只对外部调用生效）；异步线程异常不会抛到主线程，必须在 try-catch 里手动更新任务状态为 FAILED，不然任务会永远卡在 PROCESSING。"
❓ 可能追问
"轮询有什么缺点？有更好的方案吗？"
→ 缺点是浪费请求，180次轮询中可能170次拿到的都是 PROCESSING。更好的方案是 SSE（Server-Sent Events），服务端主动推送，用 SseEmitter 实现，比 WebSocket 轻量。
8
Temperature 参数
▼
🎯 通俗比喻

Temperature 就像"创意旋钮"。旋到最低（接近0），AI 像一丝不苟的会计，每次给最确定的回答；旋到最高，AI 像天马行空的诗人，回答更有创意但也更不可预测。

📖 技术本质

Temperature 控制 LLM 输出的概率分布。低 temperature 让分布变"尖锐"，高概率 token 被选中概率更大，输出更确定；高 temperature 让分布变"平坦"，低概率 token 也有机会被选中，输出更多样。数学上是在 softmax 中除以 temperature 值再归一化。

💭 我的理解
"我的意图分析 temperature 设为 0.1——因为需要确定性的结构化输出。如果设高了，同一句'画一只猫'有时被解析成 IMAGE 有时被解析成 VIDEO，系统就不稳定了。但做创意生成的 Prompt，temperature 反而应该高一些。关键是理解场景需要确定性还是创造性。"
❓ 可能追问
"除了 temperature，还有什么参数影响输出？"
→ top_p（核采样，限制候选 token 累积概率）、top_k（限制候选 token 数量）、max_tokens（限制输出长度）。意图分析中主要依赖 temperature 和结构化 Prompt 来控制。

⭐ 被问"这不就是调API吗"怎么回答

9
Spring Boot @Async 异步执行
▼
🎯 通俗比喻

@Async 就像你给员工一个任务说"你去处理这件事，完了更新到看板上，我先去做别的了"。主线程不等你，另起一个线程去干活。

📖 技术本质

Spring @Async 通过 AOP 代理实现。加了注解的方法被调用时提交到 TaskExecutor 线程池异步执行。需要 @EnableAsync 开启，返回值可以是 void 或 Future/CompletableFuture。

💭 我的理解
"踩了两个坑。第一，同类内部调用不生效——因为 AOP 是基于代理的，this 调用绕过了代理。第二，异步线程异常不会传播到调用者——必须在方法内部 try-catch，手动把错误状态写到数据库。"
❓ 可能追问
"默认线程池有什么问题？"
→ SimpleAsyncTaskExecutor 每次新建线程没有复用，生产环境要换 ThreadPoolTaskExecutor 设置核心线程数、最大线程数、队列容量。
"异步方法的事务怎么处理？"
→ 异步方法在新线程执行，和调用者不共享事务上下文，需要独立的 @Transactional。
10
OkHttp 调用外部 API
▼
🎯 通俗比喻

OkHttp 是一个专业的快递员——支持自动重连、连接池复用、代理中转，比邮局（RestTemplate）更灵活。

💭 我的理解
"选 OkHttp 不是因为'更好'，是因为场景匹配。Google API 返回 Base64 编码的二进制数据，OkHttp 的 ResponseBody.bytes() 更直接；代理配置一行 builder.proxy() 就搞定。RestTemplate 调 OneRouter 反而更合适，标准 JSON 交互没必要引入额外复杂度。"
11
RESTful API 设计
▼
🎯 通俗比喻

REST 像图书馆——GET /books 是看书单，POST /books 是新进一本书，DELETE /books/123 是下架123号书。动词靠 HTTP 方法，名词靠 URL 路径。

💭 我的理解
"generate 用 POST 因为有副作用（创建任务记录）。task/{taskId} 用 GET 因为幂等，重复调用不改变状态——对前端轮询很重要，网络抖动导致重复请求不会出问题。"
12
轮询 vs WebSocket vs SSE
▼
🎯 通俗比喻

轮询：你每分钟看一次手机有没有新消息
WebSocket：你和朋友打电话，双方随时说话
SSE：你订阅了新闻推送，有消息它主动推给你

📖 对比
特性轮询WebSocketSSE
方向客户端→服务端双向服务端→客户端
连接短连接长连接长连接
开销大量无效请求维护连接状态中等
复杂度低高中
适用状态查询实时聊天进度推送
💭 我的理解
"我选轮询因为实现简单、和 REST 一致、断线自动恢复。如果优化，SSE 最合适——任务状态推送是单向的，不需要 WebSocket 的全双工。SSE 用 SseEmitter 就能实现，比 WebSocket 轻量。"
13
策略模式 / 工厂模式在 Provider 中的应用
▼
🎯 通俗比喻

策略模式：像导航软件——同样的目的地，"最快""最短""避开高速"用不同算法。
工厂模式：像自动贩卖机——按图片按钮出图片生成器，你不需要知道里面怎么造的。

💭 我的理解
"我没刻意去'用设计模式'，而是在解决'怎么让代码好扩展'的问题时，最自然的解法就是这些模式。一开始图片和视频逻辑写在一起，每加一个模态就改很多地方。抽出接口后，加新模态就是'实现接口+注册'两步，自然就成了策略+工厂的结合。"
14
错误处理与降级
▼
🎯 通俗比喻

就像飞机有两个引擎——一个挂了还能飞。好的系统不是"永远不出错"，而是"出错了还能体面地应对"。

📖 三层错误处理

Provider 层：generate() 内部 try-catch，失败返回 failure()，不抛异常。
Service 层：检查 result.isSuccess()，失败更新 FAILED 状态。
外层兜底：大 try-catch 防任何未预期异常导致任务卡在 PROCESSING。

💭 我的理解
"AI 应用的'错误'可能是模型触发安全过滤、结果质量差、API 超时。我的原则：可重试的自动重试，不可重试的快速失败并告知用户。如果继续优化，可加熔断器（Circuit Breaker），OneRouter 挂了自动降级到规则引擎做意图分析。"

🔧 技术挑战（工程视角）
挑战1：@Async 的事务和异常陷阱
"异步方法在新线程执行，不共享事务上下文。一开始在 generate() 里创建任务后调异步方法，异步方法查任务发现查不到——因为主线程事务还没提交。后来把创建和异步调用放在事务提交后执行才解决。"
挑战2：OkHttp 超时配置
"调 Gemini 图片 API 默认10秒超时经常失败。调 Veo 视频 API 需要等几分钟。我给不同场景配了不同超时——图片 readTimeout 120秒，视频600秒。通过不同的 OkHttpClient 实例实现。"
☕ 被问"为什么用 Java 而不是 Python"
"AI 应用确实很多用 Python。但我选 Java 是因为：Spring Boot 的企业级特性（依赖注入、AOP、异步支持）让架构设计更规范；Java 类型系统对大项目可维护性更好；我也更熟悉 Java 生态。模型训练和微调 Python 更合适，但做 AI 应用的后端服务，Java 完全没问题。"
